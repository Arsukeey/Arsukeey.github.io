Functional Programming

This is a short introduction to functional programming using Elixir for examples, but you don’t need to know Elixir to understand them.

History

So, functional programming. Started in 1958 with Lisp by John McCarthy, it is a programming paradigm that’s gaining a lot of traction recently, since it makes programs more readable and easier to maintain, since it tries to treat computations as mathematical functions, causing no side effects.

Lisp was extremely important to AI development, and we probably wouldn’t be here if Lisp didn’t exist.

Lambda Calculus

The bread and butter of FP1 is Lambda Calculus. It was invented by Alonzo Church in 1930, and it consists of functions represented as black-boxes.

An example:

fn x -> x * x end # Squares argument

This concept, normally called closures or anonymous functions in other programming languages, is the core of FP.

Map, filter and reduce

Those are famous functions, implemented in almost every programming language (just like Lambda Calculus), and they apply transformations to arrays or lists.

Map - applies a function in every element of a list.

Enum.map([1, 2, 3], fn x -> x * x end) # returns [1, 4, 9]

Filter - returns a list of every element that, according to the given function, is true

Enum.filter([1, 2, 3, 4], fn x -> if rem(x, 2) == 0 do x end) # Returns only even numbers, [2, 4]

Reduce - transforms a list in a single value.

Enum.reduce([5, 10, 15, 20, 25], fn x, acc -> x + acc end) # Returns 75 (every element of the list summed)

Immutability

One of the principles of FP is immutability, and that means that variables’ values don’t actually vary. It is important because it turns functions into predictable black boxes, and as a side effect, the performance is better, since it doesn’t allows copying values around.

First-class functions

That means that functions are also treated as values and they can be used as data. That also means that they can return functions and receive functions as paramenters, making them a lot more powerful.

Higher-order functions

These are functions that either: - take functions as arguments, or - returns a function
